## 1. Зрелость фреймворка к реальному использованию и релизу

### 1.1 Что сильно держит продукт

- **Три слоя CLI → Service → Backend**, явное разделение; тестовый контур большой (тысячи pytest).
- **SENAR QG-0 / QG-2** реально проверяются в сервисе: задача без цели и AC не стартует; закрытие без верификации и гейтов блокируется.
- **MCP как основной контракт для агента** — близко к паритету с CLI; есть структурированный `tausik_task_done_v2` с разбором гейтов.
- **Dogfooding и метрики** (throughput, FPSR, DER и др.) — редкость для «методологических» фреймворков; даёт ощущение зрелой модели управления.
- **Документация по объёму и глубине** выше среднего: architecture, hooks, mcp, shared-brain, quickstart EN/RU.

### 1.2 Риски для первого впечатления «незрелого» продукта

1. **`tausik_task_done` «зависает» в VS Code Claude Extension**  
   - По коду: `asyncio.to_thread(handle_tool, …)` не блокирует event loop Python, но **весь** `task_done` (БД + цепочка гейтов + subprocess, например pytest) выполняется **синхронно до конца** одним ответом MCP.  
   - Прогресс пишется в **stderr** (`[gate i/n] …`), что **часто не показывается** в UI хоста — пользователь видит «тишину» минутами.  
   - В `gate_runner.py` у каждого гейта таймаут по умолчанию **120 с**; несколько гейтов подряд + тяжёлый pytest = **многоминутный** wall time — нормально для CI, тяжело для интерактивного MCP, если клиент ещё и обрезает время ожидания.  
   - **Вывод:** зависание часто сочетание «долгая верификация» + «нет видимого прогресса» + возможный **внешний таймаут** расширения, а не обязательно deadlock.

2. **`tausik_task_done` (v1) при ошибке QG-2 отдаёт только первое** `blocking_failure` из отчёта (`service_task.py`) — агенту сложнее чинить всё сразу; v2 лучше, но промпты по умолчанию часто ссылаются на v1.

3. **Project MCP не делает `chdir` на `--project`**, в отличие от brain-сервера; `_project_dir()` в handlers завязан на **`os.getcwd()`**. При несовпадении cwd и корня репозитория (типично для части интеграций) возможны тихие сбои путей (стеки, скиллы, относительные пути).

4. **Документация MCP расходится с реальностью:** заявленные **90+6=96** инструментов vs **91+6=97** в коде; неверные имена полей в таблице brain (EN); для stack-инструментов в доке `stack`, в схеме **`name`**; описание `verify` как с опциональной задачей не совпадает со схемой (обязателен `task_slug`). Это бьёт по доверию при «первом контакте».

5. **Git pre-commit vs «quality gates»:** в документации создаётся впечатление связи pre-commit со scoped tausik gates; фактический `scripts/hooks/pre-commit` — **mypy** (+ опционально RAG), а не вызов `tausik gates`. Нужна честная терминология.

6. **Паритет IDE:** bootstrap для **Cursor** не генерирует эквивалент `.claude/settings.json` с Python-хуками; жёсткие PreToolUse/PostToolUse в полном объёме ориентированы на **Claude Code** (и Qwen в той же схеме). Для Cursor/GPT/Composer опора в основном на **правила + MCP** — сила SENAR «принуждение важнее договорённостей» слабее без хуков.

### 1.3 Современные модели (Claude, Composer, GPT-5.x)

| Аспект | Claude Code + TAUSIK | Cursor / VS Code + другие модели |
|--------|----------------------|-----------------------------------|
| Хуки | Полный пайплайн возможен | Часто нет тех же хуков; нужна явная документация |
| MCP | Нативно, стабильнее длинные вызовы | Работает, но UX длинных tool calls хуже; важны короткие ответы и предварительный `verify` |
| Скиллы (`/plan`) | Есть экосистема slash | Нет; нужен текстовый онбординг «делай так» |
| Память `~/.claude` | Упоминается в правилах | Может сбивать; нужен нейтральный слой онбординга |

**Рекомендация для 1.4:** один короткий раздел «не-Claude»: MCP-имена инструментов, цикл `task start` → `verify` → `task_done_v2`, без отсылок к auto-memory Claude-only.

## 2. MCP-серверы (детально)

### 2.1 Состав

- **tausik-project** — основной набор (~91 handler / tool), stdio, `mcp` SDK.  
- **tausik-brain** — 6 инструментов Shared Brain; при старте **`os.chdir(args.project)`**.  
- **codebase-rag** (опционально) — 7 инструментов, вынесен отдельно.

### 2.2 Сильные стороны

- Ясная структура `tools.py` / `tools_extra.py` / `handlers.py`.  
- Brain: понятные сообщения при отсутствии токена / не настроенном brain.  
- `task_done_v2` — правильное направление для машинно-разбираемых ответов.

### 2.3 Недочёты и пробелы

| ID | Тема | Суть |
|----|------|------|
| M1 | cwd | Project MCP без `chdir` на `--project`; `getcwd()` в handlers |
| M2 | Таймауты / UX | Нет агрегированного лимита на весь `task_done`; прогресс только stderr |
| M3 | verify | MCP урезан относительно CLI: нет task-less режима, нет `scope` в API, урезанный вывод |
| M4 | Ошибки | У project server при исключениях нет traceback в stderr (brain лучше) |
| M5 | Доки | Счёт tools, brain-поля, stack `name`, verify параметры |
| M6 | task_done v1 | Только первое сообщение об ошибке из списка |

### 2.4 Почему это критично для VS Code

Расширения часто ждут **один** ответ tool call. Длинный синхронный вызов без промежуточных уведомлений в протоколе MCP выглядит как зависание. Митигации до изменений протокола: **документировать** `verify` до `done`, **scoped** тесты, **`task_done_v2`**, настройка таймаутов гейтов в `config.json`, исследование **разбиения** «закрыть задачу» и «прогнать гейты» (двухфазная модель) на 1.4+.

## 3. Shared Brain

### 3.1 Что хорошо

- Чёткая **философия разделения**: локальная БД проекта vs обобщаемые знания в Notion.  
- **Scrubbing + classifier + project hash** — продуманная модель приватности.  
- **Локальное зеркало** `~/.tausik-brain/brain.db` + FTS5 — быстрый офлайн-поиск после sync.  
- Документ `shared-brain.md` описывает архитектуру и каскад env (в т.ч. `.tausik/.env`).

### 3.2 Слабые места

- В шапке документа: **«setup wizard pending»** — высокий порог входа (Notion, интеграция, шаринг страницы, 4 БД). Для «не для галочки» не хватает **мастера** или **одной команды** `brain init` с проверками.  
- **MCP brain** удобен для записи/поиска, но **ежедневный цикл разработчика** («перед задачей подтянуть 3 релевантных паттерна») слабо зашит в скиллы по сравнению с project task flow.  
- Зависимость от **Notion API** (лимиты, 429) — нужны явные рекомендации по retry (часть уже в клиенте) и **сценарии «brain offline»** в UX.

### 3.3 Улучшения (все смыслы)

1. CLI / bootstrap: **wizard** или интерактивный чеклист.  
2. Скилл `brain`: сценарий «старт сессии → `brain_search` по стеку задачи».  
3. Метрики использования brain (хотя бы счётчик запросов в сессии) для dogfooding.

## 4. SENAR и соответствие TAUSIK

Из **SENAR-Summary-RU** (v1.3): пять ценностей (контекст, верификация, знание, принуждение, суждение), Core 8 правил, QG-0…QG-4, чеклист 28 пунктов, метрики включая FPSR и тупики.

**TAUSIK хорошо реализует:** QG-0/QG-2 как код, метрики, dead ends, сессии, чеклистная логика через гейты и verify cache.

**Зазоры методологии:**

- **Принуждение** сильно там, где есть **хуки** (Claude Code); в других IDE это в основном **инструкция** — SENAR-идея «принуждение важнее договорённостей» требует **явного** признания в гайде и компенсирующих паттернов (например обязательный `verify` в скилле `/ship`).  
- **Верификация важнее скорости** иногда конфликтует с **интерактивным MCP** (долгий `task_done`) — методологически верно, продуктово нужно **предсказуемое время отклика** или стадирование.

**Предложения к SENAR (для обсуждения в стандарте, не как факт релиза):**

- Отдельный подпункт **«инструментальные хосты»** (IDE/клиент MCP): различие hard enforcement (hooks) vs soft (prompt-only).  
- **Рекомендация по длинным верификациям:** разделение «зафиксировать намерение закрыть» и «подтвердить гейты» или явный SLA на tool call для агентских клиентов.

## 5. Как фреймворк помогает разработке (вывод)

**Помогает:** связывает работу агента с **задачей, AC, логами и метриками**; снижает галлюцинацию «я всё сделал» через QG-2; даёт **воспроизводимый** контур verify + scoped pytest; Shared Brain — правильная ось для **межпроектных** знаний.

**Мешает или рискует:** долгие синхронные MCP-вызовы без обратной связи в UI; расхождение док/кода; ожидание паритета хуков там, где его нет; порог настройки Brain.
